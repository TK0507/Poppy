section-1 大局的な目標 {

    1日目の目標を「開発目標の具体化および詳細化」に決定しました。
    現状では自然言語処理を用いたテキスト処理アプリケーションを考えています。

    大まかには以下のようなものがあります。

    1.特定のドメインに特化した自然言語処理アプリケーション
    2.任意のドメインに対応した自然言語処理アプリケーション

    ここで、1,2についてさらに検討してみます。

    1-1.特定のドメインに特化した対話用チャットボット
    1-2.特定のドメインに特化した制御用チャットボット
    1-3.特定のドメインに特化したテキスト分析ツール

    2-1.任意のドメインに対応した対話用チャットボット
    2-2.任意のドメインに対応した制御用チャットボッ
    2-3.任意のドメインに対応したテキスト分析ツール

    これらの中から、興味深い結果を得ることができ、
    短期間で実際に動作するアプリケーションを開発できるものとしては

    1-1, 1-2, 1-3, 2-3

    等であると考えましました。

    選択されたなっか候補の理由としては

    2-1 ... すでに多くのサービスが存在します。
    2-2 ... 任意ドメインにおける様々な命令文を実装することは、
        自然言語処理アプリケーション開発の経験のない私にはおそらく困難です。

    という理由です。
    ただし、これは現時点での主観的な判断であるため、再検討される可能性があります。
    少なくとも現在はこの二つを除くものとします。

    1-1, 1-2, 1-3, 2-3

    この中でより開発しやすいモデルとしては。

    1-3 = 2-3 > 1-2 > 1-1

    のような順序になると考えましました。
    その理由は、そもそもチャットボットを開発する際にはそのテキスト処理の中途過程で
    テキスト分析（分類及び予測）を実行する必要があるため、
    実際のところチャットボットはテキスト分析の延長線上に存在すると考えたからです。
    ただし、これは現時点での主観的な判断であるため、再検討される可能性があります。

    結局のところ、分類や予測タスクの分析の実装を試してみますほかないことが判明しました。
    そこで、今回の最初の目標としては「テキスト分析アプリケーション」の候補である

    1-3, 2-3

    等を選択します。

    ここで、テキスト分析アプリケーションを開発することが決定しました。
    以降上記のセクション

    決定事項: テキスト分析アプリケーション開発
}

section-2 分類の場合 {

    どのようにアプリケーションを開発するのかについての具体的な内容を検討します。

    テキストを分類する場合

    1.確率分布による方法
    2.分散ベクトルによる方法

    等の候補が考えられます。

    下記に書く方法の説明を列挙します。

    1 ... OHEなどによって排他的なラベルによってテキストを分類します。
    2 ... 排他的ではなく、直交するベクトル属性の複合によってテキストを分類(クラスタリング)します。

    両者のどちらの場合においても、テキストと分類結果の関係マップ(教師データ)が必要不可欠です。
    ただし、1の場合においては特定の分類結果を取得したい場合に2よりも精度がよくなる可能性が高く、
    逆に2の場合はより広い範囲で汎用的な分類を行いたい場合にはより適切である可能性が高いです。
    (NOTE: 2は実際には回帰タスクに分類されるのかもしれない)

    しかし、テキストが完全に単一のトピックから構成されているとは限らないという仮説を立てると、
    おそらく1のアプローチは誤った方法である可能性が高いです。
    ただし、これらは主観的な判断であるため、再検討される可能性があります。

    そのため、今回は2のアプローチをとることとします。

    決定事項: 分散ベクトルによるテキスト分類
}

section-3 回帰の場合 {

    回帰モデルを利用する場合は、必ずと言っていいほどテキスト生成が必要になると考えられます。
    しかし、今回テキスト生成はsection-1において困難であると判断したため、優先順位が低いです。
    そのため回帰タスクは実際のところ考慮するべき事項ではありません。
    しかし、一応回帰タスクをプログラムする場合のことを検討します。

    テキストを生成する場合、
    入力に対する応答としてそのアプリケーションが持つ機能的側面を踏まえた返答を要求される。
    この場合、テキスト分析の結果を再利用した方法を考慮することができ、
    制御用の場合にはテキストの詳細からコマンド及びコマンドに対するパラメータを生成し、
    実行時に適用する必要があります。

    そのため、分類タスクが安定した後にテキスト生成部分を実装するとします。

    決定事項: テキスト生成はテキスト分析依存
}
